## Operation 1: registerUser

### Contract: registerUser(name: string, email: string, password: string): UserDto

**Cross-References**: UC018 Main Flow Steps 8-12

**Preconditions**:

- `email` is not null or empty
- `email` is valid email format
- `email` does not exist in User table (including soft-deleted records)
- `password` is not null or empty
- `password` is at least 8 characters long
- `password` contains at least one uppercase letter
- `password` contains at least one lowercase letter
- `password` contains at least one number
- `name` is not null or empty

**Postconditions**:

- A new instance `u` of User was created
- `u.id` was set to a new unique CUID
- `u.email` was set to `email` (lowercase)
- `u.name` was set to `name`
- `u.password` was set to bcrypt hash of `password` (salt rounds: 10)
- `u.role` was set to UserRole.RegisteredUser
- `u.emailVerified` was set to null
- `u.image` was set to null
- `u.createdAt` was set to current timestamp
- `u.updatedAt` was set to current timestamp
- `u.deletedAt` was set to null
- A UserDto was returned containing:
  - id, email, name, role, emailVerified, image, createdAt, updatedAt
  - **excludes**: password, deletedAt

## Operation 2: checkEmailExists

### Contract: checkEmailExists(email: string): boolean

**Cross-References**: UC018 Main Flow Step 8

**Preconditions**:

- `email` is not null or empty
- `email` is valid email format

**Postconditions**:

- Returns `true` if a User with matching email exists (including soft-deleted)
- Returns `false` if no User with matching email exists
- No data was modified in the database

## Operation 3: hashPassword

### Contract: hashPassword(password: string): string

**Cross-References**: UC018 Main Flow Step 9

**Preconditions**:

- `password` is not null or empty
- `password` length is at least 8 characters

**Postconditions**:

- A bcrypt hash of `password` was computed using 10 salt rounds
- The hash string was returned (60 characters, starting with $2b$10$)
- The original password cannot be recovered from the hash
- The same password will produce different hashes (due to random salt)
- No data was modified in the database

---

## Operation 4: createSession

### Contract: createSession(userId: string): Session

**Cross-References**: UC018 Main Flow Step 11

**Preconditions**:

- `userId` is not null or empty
- A User with id = `userId` exists
- User with id = `userId` is not soft-deleted

**Postconditions**:

- A new JWT session token was created containing:
  - `sub`: userId
  - `email`: user's email
  - `role`: user's role
  - `name`: user's name
  - `iat`: issued at timestamp
  - `exp`: expiration timestamp (30 days from now)
- Session token was signed with JWT_SECRET
- Session token was returned
- Session token can be validated to authenticate future requests
- No database modifications occurred (session is stateless JWT)

## Operation 5: validateRegistrationInput

### Contract: validateRegistrationInput(input: RegisterInput): ValidationResult

**Cross-References**: UC018 Main Flow Step 8

**Preconditions**:

- `input` object is not null

**Postconditions**:

- If validation passes:
  - Returns `{ success: true, data: ValidatedInput }`
  - `data.email` is trimmed and lowercased
  - `data.name` is trimmed
  - `data.password` is validated but not modified
- If validation fails:
  - Returns `{ success: false, errors: ValidationError[] }`
  - Each error contains: field name, error message, error code
  - No data was modified

**Validation Rules Applied**:

1. `name`: required, string, min 1 character, max 100 characters
2. `email`: required, valid email format, max 255 characters
3. `password`: required, min 8 characters, max 72 characters (bcrypt limit), must contain uppercase, lowercase, and number

## Invariants

### User Invariants

1. `User.email` must be unique across all records (enforced at database level)
2. `User.password` must always be a bcrypt hash, never plain text
3. `User.role` must be one of the defined UserRole enum values
4. `User.createdAt` must be less than or equal to `User.updatedAt`
5. If `User.deletedAt` is not null, user is considered soft-deleted

### Security Invariants

1. Plain text passwords must never be stored in the database
2. Plain text passwords must never be logged
3. Plain text passwords must never be returned in API responses
4. Session tokens must expire after 30 days
5. Password hashes must use bcrypt with minimum 10 salt rounds

### Business Logic Invariants

1. All new registrations must have role RegisteredUser
2. Email addresses are case-insensitive (stored lowercase)
3. Soft-deleted user emails cannot be reused for new registrations
4. User IDs must be unique CUIDs generated by the system
